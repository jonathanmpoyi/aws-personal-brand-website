# Personal Brand Website — Deployed on AWS

A personal brand website for a cloud security engineer, built with HTML and deployed on AWS using a secure, globally distributed architecture. Live at [josephnathanmpoyi.com](https://josephnathanmpoyi.com).

---

## Architecture

```
User
  ↓
Route 53 (DNS — Alias record pointing to CloudFront)
  ↓
CloudFront Distribution (Global edge caching + HTTPS + OAC)
  ↓
S3 Bucket (Private — ca-central-1)
  ↓
index.html (Static website content)
```

| Service | Purpose |
|---|---|
| Amazon S3 | Hosts the static website files in a private bucket |
| Amazon CloudFront | Global content delivery with edge caching and HTTPS enforcement |
| AWS Certificate Manager (ACM) | Provisions SSL/TLS certificate for encrypted HTTPS access |
| Amazon Route 53 | Domain registration and DNS routing to CloudFront |

---

## Security Controls

**Private S3 Bucket**
Public access is fully blocked on the S3 bucket. The website content is not directly accessible via the S3 URL. Any direct request to the bucket returns Access Denied.

**Origin Access Control (OAC)**
CloudFront is configured with OAC, which signs every request to S3 using AWS Signature Version 4. The S3 bucket policy only allows GetObject requests from this specific CloudFront distribution. No other source can read from the bucket.

**HTTPS Enforcement**
An SSL/TLS certificate was provisioned in AWS Certificate Manager in us-east-1 (required by CloudFront) and attached to the CloudFront distribution. All traffic is served over HTTPS, ensuring encryption in transit, authenticity, and integrity.

**Bucket Policy (Auto-generated by CloudFront OAC)**
```json
{
  "Effect": "Allow",
  "Principal": {
    "Service": "cloudfront.amazonaws.com"
  },
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::your-bucket-name/*",
  "Condition": {
    "StringEquals": {
      "AWS:SourceArn": "arn:aws:cloudfront::ACCOUNT_ID:distribution/DISTRIBUTION_ID"
    }
  }
}
```

## Architecture Decisions

**S3 bucket in ca-central-1**
The primary audience is Canadian and North American. Deploying the origin bucket in ca-central-1 minimises latency on CloudFront cache misses, when the edge location needs to fetch content from the origin for the first time.

**Certificate in us-east-1**
AWS Certificate Manager certificates must be provisioned in us-east-1 to be attached to CloudFront distributions. This is an AWS architectural requirement regardless of where other resources are deployed.

**Versioning enabled on S3**
S3 versioning is enabled to maintain a full history of every file update. This allows recovery of any previous version of the website in case of accidental overwrites or data loss.

## Deployment Steps

1. Create a private S3 bucket in ca-central-1 with public access blocked and versioning enabled
2. Upload index.html to the bucket
3. Create a CloudFront distribution pointing to the S3 bucket REST endpoint as the origin
4. Enable Origin Access Control (OAC) during CloudFront setup
5. Apply the auto-generated bucket policy from CloudFront to the S3 bucket
6. Set the Default Root Object to `index.html` in the CloudFront distribution settings
7. Provision an SSL/TLS certificate in ACM in us-east-1 using DNS validation
8. Register the domain in Route 53 and create the DNS validation record for ACM
9. Add the custom domain as an Alternate Domain Name in CloudFront and attach the ACM certificate
10. Create an Alias record in Route 53 pointing the domain to the CloudFront distribution
11. Test the live website at your custom domain over HTTPS
